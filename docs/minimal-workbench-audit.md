# 极简工作台审计账本

目标：让 TTS More 像一个配音生产工作台，而不是模型配置控制台。默认路径短，异常信息明确，高级能力只在需要时展开。

## 主路径

```mermaid
flowchart LR
  A["剧本"] --> B["提取台词"]
  B --> C["角色音色"]
  C --> D["TTS 接入"]
  D --> E["生成台词"]
  E --> F["试听历史"]
```

三栏只承担三件事：

- 左侧：选择、新建、编辑、保存和提取剧本。
- 中间：查看台词、筛选、选行、批量生成、行内试听。
- 右侧：编辑当前台词的音色、参考资源、生成文本和生成动作。

顶部入口使用任务语言：

- `角色`：给当前剧本角色选择或保存音色。
- `队列`：查看生成进度和最近任务。
- `解析`：保存并检查剧本解析服务。
- `接入`：检测并保存 TTS 服务地址。

## 本轮收敛

| 区域 | 旧问题 | 当前规则 |
| --- | --- | --- |
| 空工作台 | 没有剧本或台词时仍显示筛选、生成、清除筛选和 Inspector 说明 | 默认只显示安静空态；有台词后才显示生产控件 |
| 角色库 | 没有当前剧本角色时自动展开全局常用音色详情 | 不自动选择全局角色；用户明确选择后才显示详情 |
| 参考音频 | 全量样本容易把角色详情变成资产清单 | 默认只预览少量样本，并提示还有多少可在详情中选择 |
| 行卡 | 未生成行也显示“暂无音频”一类诊断 chip | 普通行卡只保留角色、括注、台词、状态、试听/生成；异常和历史才显示补充 chip |
| Inspector | load signature、provider、binding 等技术字段过早出现 | 正常状态折叠；异常、队列中或用户展开诊断时出现 |
| LLM 面板 | 主动作写成“保存并激活”，容易让用户以为会触发复杂模式 | 主动作改为“保存并检查”；测试连接保留为诊断动作 |
| TTS 接入 | 旧地址文案不利于 worker-first 架构 | 主文案改为“服务地址”；worker 和 Gradio 都按服务端点理解 |
| Toast | 启动时重复堆叠“就绪” | 被动 ready 状态只留在顶部状态，不进入 toast 栈 |

## 设计规则

- 默认界面只显示下一步可做的事。
- 没有输入时不显示生产控件；没有异常时不显示诊断实体。
- 能用用户任务命名时，不用内部对象命名。
- `profile`、`binding_id`、`cluster key`、`load signature` 只出现在诊断、历史或展开详情中。
- 文档面向人和 Agent 同时阅读：先写目标、输入、输出、失败处理和验收，不把推理链写成操作说明。

## Agent 读取方式

Agent 遇到工作台相关任务时按这个顺序判断：

1. 用户是否在主路径上：剧本、台词、角色音色、TTS 接入、生成、试听历史。
2. 当前是否缺输入：没有剧本、没有台词、没有音色或没有服务地址。
3. 是否存在异常：服务不可达、资源缺失、生成失败、队列阻塞。
4. 只有异常或用户要求诊断时，才展开内部字段。

不要为了“解释清楚”而在默认 UI 或 prompt 里加入更多实体。优先删除、折叠或改名。

## 验收

- 定向测试：`pnpm --dir frontend test -- src/lib/workbenchView.test.ts src/lib/roleLibraryView.test.ts src/lib/toast.test.ts src/i18n.test.ts`
- 全量前端：`pnpm --dir frontend test`、`pnpm --dir frontend build`
- 后端发布守卫：`.venv\Scripts\python.exe -m pytest backend/tests/test_release_governance.py -q`
- 浏览器 QA：桌面 1280x720、移动 390x844；确认无横向溢出、无相关 console error、空工作台无生产控件。

## 暂缓事项

- 删除兼容 i18n key：旧 key 可能仍被旧数据或隐藏兼容分支引用，主路径不再展示即可。
- 前端拆包：Vite 只有 chunk-size warning，作为单独性能任务处理。
- 真实 GPU 验收：仍需可用模型和 GPU 环境，见 `docs/ci-architecture.md`。
